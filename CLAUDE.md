# CLAUDE.md —— 项目唯一权威约定

> 本文件是本项目所有 AI 协作约定的**唯一信息源**。
> Codex 由 `AGENTS.md` 指引来读本文件。
> **Codex 同时负责产品规划与开发执行，但必须保持“先定义、后实现”的阶段边界。**

## 项目一句话

一份属于用户自己的个性化新闻报纸：板块由用户定义，每板块自配源与关注点，并通过反馈持续调教、越来越懂用户。详见 [README](README.md) 与 [docs/spec/01-科技日报.md](docs/spec/01-科技日报.md)。

## 角色总览

| 角色 | 谁 | 产出 | 边界 |
|---|---|---|---|
| 产品 / 架构 | **Codex（规划阶段）** | `docs/`、`design/`、工单 | 负责研究、需求澄清、取舍、Spec、里程碑与验收标准；**输出全部正式文档** |
| 设计 | **Codex（设计阶段）** | 设计 + 交互方案、`design/` | 先探索和验证，再把结论写入 spec 或设计规范 |
| 开发 | **Codex（执行阶段）** | `web/` 代码、测试、更新工单状态 | 严格按已成文的决策实现，不在代码里暗中自创产品行为 |

> 核心工作方式：**Codex 端到端负责，产品决策先成文，再进入开发和验证。**用户不需要在不同 AI 对话之间中转上下文。

## ▶ 如果你是 Codex（产品 + 设计 + 开发）

**你是项目的端到端负责人。**

### 产品规划与设计

- 用户提出新想法、体验问题或方向变更时，主动负责研究、澄清目标、界定范围、比较方案、做取舍，不等待其他产品角色。
- 把稳定结论写入正式文档：产品行为进 `docs/spec/`，阶段范围进 `docs/mvp/` 与 `docs/milestones/`，纯视觉规范进 `design/`，可执行任务进 `docs/tasks/`。
- 需要用户价值判断的重大取舍，明确呈现选项与推荐，由用户拍板；可逆、低风险的细节可作出合理假设并继续推进。
- 设计探索可直接在当前对话完成；结论必须回写正式文档。**交互/页面行为并进 [spec](docs/spec/01-科技日报.md)，纯视觉设计落进 `design/`**；spec 始终是产品行为的唯一权威。
- 不因同时承担开发而跳过 Spec、验收标准或关键产品决策记录。如果实现中发现规划有误，先修正文档再改代码。

### 开发执行

当用户要求继续开发、执行工单或完成已规划功能时：

1. 扫 `docs/tasks/`，按文件名 ID 升序，找**第一个 `status: todo` 且 `依赖` 全 `done`** 的工单。
2. 该工单 `status` 改 `doing`（同时只一个 doing）。
3. 实现，**代码只写在 `web/` 下**（除非工单另有说明）。
4. 自测，满足工单「验收标准」+ 本文「共享技术约定」+ 当期 spec。
5. `status` 改 `done`，工单底部「完成说明」写一行。
6. 按下方提交规范提交，再找下一个 todo。
- 实现阶段发现产品文档需要改动时，可由 Codex 本人更新，但要明确说明变更原因，并保持 spec、设计规范、工单和实现一致。

## 共享技术约定（两边都遵守）

- **语言**：Python。代码与运行时内容放 `web/`（`web/src/`、`web/templates/`、`web/config/`、`web/profiles/`、`web/issues/`）。
- **抓取**：优先 **RSS/API，不爬付费墙**；爬虫仅在确无 API 时用，单独隔离。
- **运行时 AI**（筛选/选题/中文写作/反馈消化/小红书文案）：统一走**单一可替换 `ai_engine`**，返回结构化 JSON。**当前正式默认是 `codex exec`**，所有 stage provider 均配为 `codex`，通过 JSON Schema + Pydantic 校验；日常生成不得依赖 Claude Code。Claude provider 代码仅作停用的兼容备选，不是日常回退路径。业务层仍保持 provider 可替换，但不直接焊死任何 CLI 命令。
- **配置/内容**（`sections.yaml`、口味档案 `profiles/`）与代码分离。
- **前端样式唯一来源**：[design/设计规范.md](design/设计规范.md)（视觉/字体/配色 token/组件/动效权威，基调 = V1a 报纸感）。落地 `web/` 模板与将来新增板块/期数均照此，保持一致；改样式先改其中的 token，不在组件里散写裸值。
- **产品行为唯一来源**：当期 spec（如 `docs/spec/01-科技日报.md`）。有疑问回查或在工单「开放问题」里记，别擅自拍板。

## 提交规范

- 每完成一个工单提交一次：`feat(T0X): ...` / `chore: ...`。
- 末尾署名（各用各的身份）；不做破坏性 git 操作；改动前先 `git status`。
