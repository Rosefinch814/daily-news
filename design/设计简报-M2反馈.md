# 设计简报 —— M2 反馈交互（给设计对话开场用）

> 这是 **M2 调教闭环**前端交互的设计起点。产品对话已锁定**机制/架构**（见 [spec §9](../docs/spec/01-科技日报.md)）；本简报把"需要设计对话定的交互/视觉问题"打包好。
> 产出方式：设计对话讨论清楚**交互 + 视觉**；**结论回到产品对话成文**——交互/页面行为回流 [spec §14](../docs/spec/01-科技日报.md)，纯视觉进 [设计规范.md](设计规范.md)，然后产品对话补出工单 **T04（前端反馈采集）**。
> ⚠️ 本简报**只管反馈交互**，不重开 v1 版面（V1a 报纸感已定稿，见设计规范）。

## 1. 背景（机制已定，设计别改这些）

M2 要做调教闭环：读完报纸给反馈 → 反馈进 Supabase → pipeline 消化进偏好档案 → 下一期越来越懂你。前端只负责**采集反馈并写库**，已定的硬约束：

- **采集两处**：① 每条新闻后一个**极小入口**（👍 好 / 👎 不好 / 补充说明一句话）；② 报纸**末尾一个整期集中留言区**（对整期的一句话结论）。
- **直接写 Supabase**：用 **anon key + RLS**（前端只能 INSERT），**不导出文件、不要点击之外的搬运动作**。
- **定位**：每条文章已有稳定标识（`source_item_ids` + 头条/速览 level + index），点哪条系统知道。
- **不破坏 V1a 报纸感**：反馈入口必须"极小、克制"，不能抢正文、不能把报纸搞成 App 表单。

## 2. 必须遵守的铁律（影响设计）

- **📰 报纸感优先**：V1a 基调（[设计规范](设计规范.md)）。反馈件是"批注"，不是主角——默认收敛/弱化，需要时才显形。
- **🔁 即点即写**：点 👍/👎 应有即时反馈感（已记录），但**无需刷新、无需跳走**；写库是后台静默完成。
- **🧾 诚实**：反馈不改变当期内容（当期已生成）；它影响的是**下一期**。设计上别误导用户以为点了当场就变。
- **📱 响应式**：桌面优先，断点沿用 v1（820/520）；手机上反馈入口也要好按（这正是引入后端、能在手机上反馈的意义）。

## 3. 待设计对话定的"交互"问题（重点）

1. **每条的反馈入口形态**：一个 `反馈 ▾` 折叠？三个小图标常驻？hover 才现 / 一直在？点开后 👍/👎 + 一句话输入框怎么排，才够小又好点？头条和速览（两栏密排简讯）入口要不要不同处理？
2. **微交互**：点 👍/👎 后的态——高亮/变色/打勾？能否取消或改选？补充说明是必填还是可选、什么时候出现输入框？写库成功/失败要不要轻提示？
3. **末尾整期留言区**：长什么样——一个文本框 + "提交"？要不要显示"你本期标记了哪些"（H2👎 / B5👍 …）做个小回顾？留言区和归档导航（也是 M2）在版面上怎么排序？
4. **离线/重复**：同一条反复点、或网络失败，交互上怎么兜（本地暂存？禁用重复提交？）——给个设计取向即可，实现细节工单再定。
5. **空状态/首次**：用户从没给过反馈时，入口是否需要一点点引导，还是纯靠图标自解释？

## 4. 待设计对话定的"视觉"问题

- 👍/👎/补充 入口的尺寸、图标风格、颜色（在报纸感里如何不突兀，是否沿用朱红签名色做"已选"态）。
- 折叠/展开、输入框、提交按钮的组件样式（应落进 [设计规范](设计规范.md) 的 token/组件，不散写裸值）。
- 末尾留言区的版面分量（多大、与页脚/归档导航的关系）。

## 5. 产出去向

- **交互/页面行为** 结论 → 产品对话回流 [spec §14](../docs/spec/01-科技日报.md)（反馈交互那一行/新增条目）。
- **纯视觉**（入口样式/组件/颜色）→ 产品对话整理进 [设计规范.md](设计规范.md)。
- 之后产品对话补 **T04 前端反馈采集** 工单，交 Codex 落地（POST 走 Supabase anon key，定位用 `source_item_ids`+level+index；service key 绝不进前端）。
