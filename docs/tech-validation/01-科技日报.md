# 技术验证 —— 01·科技日报

> **本期（01·科技日报）技术验证，对应 [MVP](../mvp/01-科技日报.md) 的最大技术风险。**
> 记录最大技术风险、验证方法与结论。会逐步更新。

## 最大技术风险

**AI 能否按"关注清单 + 口味档案"，从一堆科技新闻里筛出"我想看的"，并随反馈变准。**

抓取（RSS/API）、渲染（HTML）都是成熟技术，风险低。真正不确定、决定产品成败的，是 **"精准 + 可调教"** 这两件事能否用我们选的方案做到。

## 待验证的关键假设

| # | 假设 | 怎么验 |
|---|---|---|
| A | **选定的 AI 调用方式**能稳定返回**结构化 JSON**（相关度/摘要/翻译/归类）。⚠️ 调用方式待定（`claude -p` 无头模式为当前候选，API 等仍在讨论），引擎须封装为可替换函数 | 选定方式后，写最小脚本喂 3–5 条新闻，看输出是否可解析、稳定 |
| B | AI 按**关注清单**筛选，结果"基本是我想看的" | 跑一期，人工逐条核对：该来的来了吗？噪音多吗？ |
| C | 把**口味档案**喂给 AI，能让筛选**朝反馈偏移** | 给一条"多看X少看Y"反馈 → 更新档案 → 重跑 → 对比前后 |
| D | 优先 RSS/API 的源，**抓取稳定**、正文可提取 | 逐源测试可达性与正文提取质量 |

## 验证方法（M1 跑通后）

1. `python .../main.py` 生成 `web/issues/<今天>.html`，浏览器打开。
2. **精准核对**：对照关注清单逐条问"这是我想看的吗 / 有没有该来没来的"。
3. **调教核对**：给一条"多看 X / 少看 Y"的反馈 → 重跑 → 看下一期是否朝偏好偏移。
4. **引擎核对**：确认选定的 AI 调用方式（当前候选 `claude -p`）输出可被脚本稳定解析（失败率、格式漂移）。

## 2026-06-23 Checkpoint MVP 实测

> 本轮先不接自动 AI 引擎（调用方式待定，`claude -p` 为当前候选），由 Codex 代替 AI 阶段，目的是验证数据链路、文件结构、页面渲染和人工可读性。详细开发 runbook 见 `web/DEVELOPMENT.md`。

实测 run id：

```text
tech-2026-06-23-165511
```

实际流程：

```text
RSS 抓取
  -> 本地轻预筛
  -> Codex/AI 语义粗筛
  -> 全量正文补全
  -> Codex/AI 最终选题
  -> Codex/AI 中文日报结构
  -> 静态 HTML 渲染
  -> 人工打开确认
```

关键结果：

| 阶段 | 文件 | 本轮结果 |
|---|---|---|
| 抓取 | `01_raw_items.json` | 119 条有效新闻，0 失败 |
| 本地轻预筛 | `02_candidates.json` | 60 条 AI 输入池 |
| Codex/AI 粗筛 | `02_codex_shortlist.json` | 17 保留 + 10 备选 + 33 丢弃 |
| 正文补全 | `03_enriched_candidates.json` | 27 条候选全部抓到正文，0 失败 |
| 最终选题 | `04_selection.json` | 5 条头条，12 条速览 |
| 日报结构 | `05_issue.json` | 校验通过 |
| HTML 渲染 | `web/dist/` | `index.html`、`latest.html`、`issues/2026-06-23.html` |

本轮形成的技术结论：

- 纯规则粗筛不适合作为最终粗筛；规则应只做轻预筛，把候选压到约 60 条，再交给 AI 做语义判断。
- AI 粗筛需要处理英文理解、聚合快讯拆解、规则误判纠偏、同事件合并。
- 已经过 AI 粗筛后，正文补全应默认抓取全部 `keep + maybe` 候选，不应只抓前几条。
- 正文补全只应更新正文内容，不应再次用规则排序导致 AI 选中的候选被丢弃。
- v1 当前是纯静态 HTML 产物，公网访问不需要后端接口；数据库可暂缓，后续用于生成侧留档、反馈、口味档案和复盘。
- 本轮 HTML 已人工打开确认“日报还可以”，可继续进入自动化流程设计。

需要继续验证：

- 用选定的 AI 调用方式（待定，当前候选 `claude -p`）替代 Codex 手工生成 `02_codex_shortlist.json`、`04_selection.json`、`05_issue.json` 后，结构化 JSON 是否稳定。
- 连续多天运行后，RSS 源稳定性、正文提取失败率、选题偏好是否稳定。
- 手机端视觉需要继续截图验收，当前只完成 HTML 结构断点检查。

## 2026-06-24 AI 脚本化与一键流程进展

> 本轮目标：把最不可控的 3 个 AI 环节先脚本化验证，再把已验证的手动 checkpoint 串成可一键执行的 pipeline。默认仍不写 Supabase，避免调试数据污染云端。

实测 run id：

```text
tech-2026-06-24-101218
```

当前自动化命令：

```bash
cd web
./.venv/bin/daily-news run-pipeline \
  --section tech \
  --date YYYY-MM-DD \
  --ai-shortlist-provider codex \
  --ai-select-provider codex \
  --ai-compose-provider codex
```

阶段级 AI provider 已支持单独指定：

| 阶段 | task_type | 默认 provider | 可覆盖参数 |
|---|---|---|---|
| AI 语义粗筛 | `semantic_shortlist` | `codex` | `--ai-shortlist-provider codex|claude` |
| AI 最终选题 | `selection` | `codex` | `--ai-select-provider codex|claude` |
| AI 中文写作 | `issue_compose` | `codex` | `--ai-compose-provider codex|claude` |

provider 优先级：

```text
CLI 阶段参数 > web/config/pipeline.yaml 的 stage_providers > ai.default_provider
```

本轮形成的工程结论：

- 正式 AI 阶段不再把大段候选 JSON 直接塞进 prompt，而是让 AI 读取本地文件路径；这避免 prompt 过长，也保留 60 条候选规模，降低漏新闻风险。
- 三个 AI 阶段已拆成脚本：`ai-shortlist`、`ai-select`、`ai-compose`；后续完整自动化复用同一套 prompt、schema、repair 和日志逻辑。
- 新增 `run-pipeline` 一键编排：抓取 → 本地预筛 → AI 粗筛 → 正文补全 → AI 选题 → AI 写作 → 导出前端数据。
- 新 run 的关键阶段产物写入 `web/runs/<run_id>/outputs/`；AI prompt、raw、attempts、provider events、metrics、pipeline 日志写入 `web/logs/<run_id>/`，不再和产物混放。
- 旧 run 仍可读取：优先读 `outputs/`，没有时回退到旧的 run 根目录文件。
- 新增 `clean-run --pack-logs`：可把旧过程日志移到 `web/logs/<run_id>/legacy/` 并打包，不删除关键阶段 JSON。

AI provider 对比结论：

- `codex` 在本轮粗筛、选题、写作中表现更稳定，最终采用 Codex 结果生成 `05_issue.json`。
- `claude` 在粗筛/选题测试中多次触发 repair，且用户当前额度不足；可继续作为单阶段备选，但默认不作为全流程 provider。
- `claude` repair 会明显增加 token 消耗，后续如果继续测试 Claude，应优先降低 repair 次数或单独跑某一阶段对比。

本轮清理结果：

- 已执行 `daily-news clean-run --run-id tech-2026-06-24-101218 --pack-logs`。
- 旧 AI 过程文件已从 `web/runs/tech-2026-06-24-101218/` 移到 `web/logs/tech-2026-06-24-101218/legacy/`。
- 关键阶段产物已归拢到 `web/runs/tech-2026-06-24-101218/outputs/`。
- 日志包位置：`web/logs/tech-2026-06-24-101218/archive/tech-2026-06-24-101218-logs.tar.gz`。
- `web/logs/` 已加入 `web/.gitignore`，不会进入公开站点或 GitHub。

仍需继续验证：

- 用 `run-pipeline` 跑一次完整真实日报，确认 `outputs/` 新目录结构下所有阶段能端到端通过。
- 连续多天观察 AI token、耗时、正文失败率和最终日报质量。
- 后续如果要无人值守，再讨论失败重试、通知、定时触发和是否写 Supabase。

## 技术选型（与假设对应）

- AI：封装为可替换引擎函数，要求 JSON 输出；当前已支持 `codex exec` 与 `claude -p` 两种 CLI provider，并支持按 AI 阶段切换；先本地轻预筛降噪，再由 AI 做语义粗筛、最终选题和中文写作。
- 抓取：feedparser(RSS) + httpx + trafilatura/readability(正文)。
- 渲染：Python 导出 `web/dist/data/` JSON + 前端壳 HTML，由 `frontend/app.js` 客户端渲染。
- 配置：PyYAML。
