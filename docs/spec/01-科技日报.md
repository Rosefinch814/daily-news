# Spec —— 01·科技日报

> 产品规格说明，研发应能照此实现而无需自行发明产品行为。会逐步更新。
> **基于 [MVP](../mvp/01-科技日报.md)（目标）派生，给出完整产品功能与方案。**
> 目标 / 验收 / 非目标见 [MVP](../mvp/01-科技日报.md)；做事步骤见 [里程碑](../milestones/01-科技日报.md)。

## 1. 问题 & 目标用户

**目标用户**：项目作者本人（单用户、自用）。

**问题**：用户每天被动刷信息流，① 会**漏掉**真正关心的内容；② 内容"别人喂什么看什么"，**不精准、不属于自己**。

**用户想要**：不是又一个新闻聚合器，而是 **一份属于自己的报纸**——板块由自己定义、每板块自己决定关注什么、并能通过反馈**不断调教、越来越懂自己**。核心难点是 **精准 + 越来越懂我**。

## 2. 核心概念：报纸 = 板块的集合

```
板块 = 名字 + 数据源 + 关注清单(种子) + 口味档案(会长大) + 反馈入口
```
- **关注清单**：第一天的你（手写种子，AI 筛选的初始尺子）。
- **口味档案**：被调教后的你（活的偏好记忆，每板块一份，互不污染）。
- 加新板块 = 多摆一个配置块，骨架不动。

本期只落"科技"一个板块。

## 3. 产品原则（铁律）

1. **🔑 全程中文可读**：绝不需要读英文原文；英文源一律由 AI 给中文，原文链接只作备查。
2. **🚫 忠于原文**：摘要/精读/翻译只能基于原文事实，不加料、不脑补；AI 的判断（如影响分析）必须**明确标注**。
3. **精准优先**：宁缺毋滥，要"我想看的"，不要大而全。
4. **分层**：每期 头条精读 + 速览。

## 4. 版面结构

```
报头（刊名 Tourbillion News + 板块英文副标题 Technology / 日期）
  ↓
📝 今日导读（AI 一段当天综述，先给全局）        ← M2，v1 暂不做
  ↓
⭐ 头条精读 3-5 条（编号 头条1…）
  ↓
⚡ 速览 10-15 条（编号 速览1…）
  ↓
📡 关于本期（抓了几源/几条 → 留几条，透明建信任） ← M2，v1 暂不做
  ↓
💬 反馈：每条 👍/👎/备注 + 末尾整期留言区（点击式，存 Supabase） ← M2，v1 暂不做
  ↓
← 上一期 · 📚 归档 · 下一期 →                  ← M2，v1 暂不做
```

> **v1（精简验证版）只做：报头 + 头条精读 + 速览。** 标「M2」的四项（今日导读 / 关于本期 / 反馈区 / 归档导航）推到 M2——**v1 只验证"精准筛选 + 全中文可读"，调教那一整套（反馈区＋口味档案＋消化反馈）整体归 M2。**

## 5. 卡片字段

| 字段 | 速览 | 头条 |
|---|---|---|
| 编号 | ✅ | ✅ |
| 中文标题 | ✅ | ✅ |
| 中文摘要 | ✅ 短(2-3句) | ✅ |
| 〔展开〕中文精读 | —（v1 不给，保持"扫一眼"） | ✅ `<details>` 手风琴内联展开 |
| 📊 影响（AI 分析，标注） | — | ✅ |
| 来源 · 原文↗（备查） | ✅ | ✅ |

**头条分块结构**（事实与 AI 判断泾渭分明）：
```
[头条1] 中文标题
摘要：一两句把事说清。（事实）
精读：来龙去脉讲透——发生了什么、关键信息、背景。（事实，忠于原文）
📊 影响（AI 分析）：对竞争格局/你关注的公司可能意味着…（← 标清是 AI 判断）
来源 综合 The Verge、Reuters · 原文↗
```

## 7. 筛选逻辑（精准在这发生）

```
每条新闻 → AI 拿「关注清单」打两个分（v1）；M2 起 +「口味档案」一起读：
   · 和你的相关度    · 事件重要度
        ↓
命中"不想看" / 太不相关 → 丢弃
相关                    → 速览
相关 且 重要            → 头条（3-5 名额，两分都高者胜出）
```
- **同源合并**：同一事件多源都报时，AI 在筛选时顺手捏成一条（标"综合 X、Y"）。
- **跨期去重（M3 提前落地，见下）**：同源合并只管"当期同一事件"，管不了"今天和昨天重复"。后者用**两层**处理——本地确定性硬去重（预筛，零 AI）+ 选题级软去重（AI 读近几天已发索引判重），详见 §7.5。

### 7.5 跨期去重（两层：本地硬去重 + 选题级软去重）

> 解决"今天的头条/速览和昨天重复"。分工原则：**能规则化的交本地、要判断的交 AI，且判断层绝不硬删**。第一层在预筛用确定性规则把"同一条原文/同一篇转载"直接滤掉；第二层把"最近几天讲过什么"喂给 AI 选题，让它自己判"换源转述 / 无新增事实"该不该再上头条。

**第一层 · 本地硬去重（预筛，零 AI 零依赖）**
- **位置**：并进**本地预筛**（复用现有跨条目去重路径），不单开 pipeline 阶段；不改阅读链路、不引依赖、不用 AI。
- **历史窗口**：回看最近 N（默认 **7 天**，可配 `dedupe.history_lookback_days`）已发期的 `web/runs/issues/<issue_id>.json`（`load_issue`），取其头条/速览的 `url` 集合；**排除当期自身**；按日期窗口取、容忍非每日空档。
- **URL 去重（核心）**：候选 `url` ∈ 历史 URL 集 → 直接丢。比对前对 URL 做轻量 canonical 化（去常见 tracking 参数、统一 http/https、去尾斜杠）。URL 完全相同才丢——**零误判**。
- **标题归一化 hash（可选，`dedupe.title_hash_enabled`）**：治"同一篇被不同 URL 重发/转载"。小写 → 去标点 → 去媒体前缀 → 去常见噪声词（英 reportedly/says/launches…、中 独家/首发/据悉…）→ hash；命中历史标题 hash 集 → 丢。
- **聚合稿降权**：预筛对早报/晚报/8点1氪/热点导览/TOP 3/快讯等**聚合合集类**加重降权并封顶分数，压低其进入 AI 的概率（不硬删，仍可凭高相关度幸存）。
- **向后兼容**：无历史期 / 关掉开关时安全降级，行为同今天。

**第二层 · 选题级软去重（AI 判断，绝不硬删）**
- **已发索引**：选题前生成一份最近 M（默认 **3 天**、最多 **40 条**，可配 `selection_history.*`）已发头条/速览的**极简索引**（只含日期/层级/标题/来源，**不含正文**），作为制品存档并把路径注入选题 prompt。
- **AI 用法（写进选题 prompt）**：候选若只是索引里已讲事件的**换源转述 / 聚合合集 / 无新增事实**重复 → 不得选为头条，优先丢弃、必要时只放速览；候选**确有新增事实** → 必须在 `reason` 里说明"新增事实是什么"、按**跟进**处理，别把昨天讲过的主体重新包装成新头条；聚合稿默认不上头条，除非含索引未覆盖的重大新事实。
- **安全铁律**：第二层是**软去重**——只影响选题/分层（不上头条 / 降到速览 / 丢弃 / 标跟进），由 AI 判断，**绝不在本地硬删**。硬删只允许第一层的 URL / 标题 hash 完全命中。**误删一条真新闻 ≫ 漏去一条重复。**

> **仍未做（确认还有痛点再开工单）**：本地 stdlib 字符相似度（`difflib` + 字符级 n-gram Jaccard）+ 规则化事件指纹的 `near_duplicate`/`possible_followup` 分级——当前判重靠"已发索引 + AI 判断"完成，若发现 AI 判不准的换源换稿漏网，再补这一档。

## 9. 调教闭环（灵魂）〔M2〕

> 产品的灵魂：读完报纸给反馈 → 反馈被消化进偏好档案 → 下一期越来越懂你。v1 只证明"靠关注清单筛得准 + 全中文"；本章是 M2 的方案。
> 〔2026-06 更新〕反馈由旧的"方案 A 纯文本 / 无后端"改为**点击式 + Supabase 轻后端**——这正是里程碑给数据库留的"反馈 / 口味档案"用途。具体**交互形态与视觉待设计对话定稿后回流**（见 §14）。

### 9.1 反馈采集（点击式）
- **每条新闻**后一个极小入口：👍 好 / 👎 不好 / 补充说明（一句话）。
- **报纸末尾**一个整期集中留言区：对整期的一句话结论。
- 反馈直接 POST 进 Supabase，**不导出文件**。文章定位用稳定的 `source_item_ids` + level + index。
- 交互的具体形态 / 位置 / 微交互与视觉 → 设计对话定稿，回流 §14 + `design/`。

### 9.2 存储（Supabase，零 serverless 函数）
- 静态站点用 Supabase 自带 REST API 直接收反馈，**不需要写 serverless 函数**。
- **安全（铁律）**：前端只用 **anon key** + **RLS（`feedback` 表仅允许匿名 INSERT，禁止读 / 改 / 删，且不暴露其他表）**；`SUPABASE_SERVICE_ROLE_KEY` 只留服务端（pipeline）用，**绝不进前端**。
- `feedback` 表关键列：`issue_id` / `issue_date` / `section_slug` / `scope`(article\|issue) / `article_level` / `article_index` / `source_item_ids` / `signal`(up\|down\|null) / `note` / `created_at` / `digested_at`（标记是否已消化，避免重复吃）。

### 9.3 偏好档案（三档，每板块一套活文档）
| 档案（`web/profiles/<板块>/`） | 管什么 | 注入阶段 | 谁能改 |
|---|---|---|---|
| `taste.md` | 选题偏好（多看 / 少看：主题·公司·人物） | 粗筛 / 选题 | AI 增量 + 用户可手改 |
| `style.md` | 写作偏好（语气 / 长度 / 翻译要求） | AI 写作(compose) | AI 增量 + 用户可手改 |
| `seed-suggestions.md` | 对种子关注清单的**待确认建议** | （用户手动应用到 `sections.yaml`） | **仅 AI 建议，用户拍板** |

三者与**关注清单**（`sections.yaml`，硬边界、用户拥有）分离：关注清单 = 第一天的你，taste/style = 被调教后的你。

### 9.4 消化（`digest_feedback` 阶段）
- **先归并**：`feedback` 是只追加事件日志，同一条文章可能有多行（👍↔👎 改主意、取消＝`signal:null`、表态后补 note 各一行）。消化前按文章定位键聚合、按 `created_at` **取最新态**（最新非空 `signal` 为准、最新为 null＝撤回；note 取最新非空）后再喂 AI——**不让矛盾的多行各算一次**。被归并的旧行一并标记已消化。
- AI 阶段：读未消化反馈（按 `issue_date` 分组、已归并）+ 对应那期日报 + 现有 taste/style → **按类型分流增量更新**：选题 → `taste.md`、写作/翻译 → `style.md`、硬性想看 / 不看 → `seed-suggestions.md` → 涉及反馈标记 `digested_at`。
- **触发**：既是 `run-pipeline` 起手阶段、又能独立 `digest-feedback` 命令单跑；支持攒几天**批量**消化。**定时（cron）归 M4**。
- **规则与安全边界**：增量改、保留旧偏好、按日期对回新闻、消化后标记；反馈**只喂偏好档案，绝不让 AI 重写 prompt 模板**；种子清单只出建议、不自动改。

### 9.5 如何生效（注入）
- **注入 = 每次生成都发生**：粗筛/选题读最新 `taste.md`、写作读最新 `style.md`，塞进各自 prompt。档案一变下次生成立刻生效，**不改代码、不改 prompt 模板**。
- 关注清单管"能不能来"（硬边界），偏好档案管"权重 / 头条还是速览 / 怎么写"（软偏好，会长大）。
- **不训练 / 微调模型**，靠"反馈 → 偏好档案 → 越来越准"的记忆闭环。档案是人可读活文档，用户随时能看"它以为我喜欢啥"、消化歪了可手改。

## 10. 用户流程（数据流）

```
配置(源 + 关注清单)
 → [抓取] 各源新闻 + 正文提取
 → [去重·硬] 本地比对近 7 天已发期：同 URL（可选 +标题 hash）硬过滤 + 聚合稿降权（零 AI，§7.5）〔M3 提前〕
 → [筛选] AI 读「关注清单」打分(相关度+重要度) → 分层(丢弃/速览/头条)，同源合并；
          并读近 3 天已发索引做软去重(换源转述不上头条/标跟进，绝不硬删，§7.5)   〔M2 起 +口味档案〕
 → [加工] 全中文：摘要 / 头条精读 / 标注的影响分析 / 翻译                        〔今日导读：M2〕
 → [渲染] 生成当天 HTML 电子报（头条、速览）                                   〔源面板/归档导航：M2〕
 → [阅读] 浏览器打开，全程中文        ← v1 到此为止
 ┄┄┄┄ 以下整段 M2 ┄┄┄┄
 → [反馈] 每条 👍/👎/备注 + 末尾整期留言（点击式 → 存 Supabase）
 → [调教] digest 按类型分流消化 → 更新 taste/style（建议进 seed-suggestions）→ 下一期更准 ⟲
```

## 11. 数据输入与输出

**输入（v1）**：数据源清单（RSS/API，按板块）、关注清单（公司/主题/人物/不想看）。〔M2 加：偏好档案 taste/style、Supabase 里的用户反馈〕
**输出（v1）**：当天 HTML 电子报。〔M2 加：更新后的 taste/style、待确认的 seed-suggestions、历史归档〕

## 12. 开发方案（高层架构）

- **语言**：Python。代码/运行时内容放 `web/`。
- **AI 引擎**：封装成**单一可替换函数**（`ai_engine`），返回结构化 JSON。⚠️ **具体调用方式待定**——`claude -p`（Claude Code 无头模式）是当前候选，但 API 等其他方式仍在讨论，调用入口须保持可替换。
- **抓取**：优先 RSS/API，不爬付费墙。
- **渲染（前后端分离·仍是纯静态）**：Python 管道产出**数据层** JSON（`data/manifest.json` + `data/issues/<日期>.json`），**表现层**为轻量壳 HTML + 共享前端 JS（`frontend/`）客户端渲染。**无服务器**，可直接静态托管（Netlify `_redirects` 兜底）。本地 `file://` 打开时另产 `.js` 兜底（fetch 跨域失败时回退 `window.DAILY_NEWS_*`）。
- **运行**：手动跑命令生成。
- 组件（建议）：fetch（抓取+正文）/ ai_engine（筛选+生成）/ render（导出数据 JSON + 构建前端壳）/ frontend（客户端渲染 app.js/app.css）/ main（编排）。〔M2 加：前端反馈采集 + Supabase 存储 + digest_feedback 消化分流；注意 M2 起静态站点首次引入 Supabase 轻后端（仅收反馈，阅读链路仍纯静态）〕
- ⚠️ **具体技术栈细节（库选型、目录布局）待最终确认**（用户要求最后再定）。

## 14. 前端交互/功能（设计对话已定 · 已回流）

页面"长什么样"见 §4、§5；"**页面怎么动**"（静态无后端下的交互）在设计对话已讨论定稿，结论如下。视觉/组件细节见 [design/设计规范.md](../../design/设计规范.md)；**v1 渲染目标 = 高保真原型 [design/prototype/v1-科技日报.html](../../design/prototype/v1-科技日报.html)**（Codex 照此落地 web/）。

| 交互项 | 定论 |
|---|---|
| 展开精读 | **`<details>` 手风琴内联展开**，零依赖、原生可访问、不跳走。头条专属。 |
| 速览"按需精读" | **不给精读**——速览只 `编号 + 标题 + 一句话短摘要 + 来源·原文↗`，保持"扫一眼"定位、降低生成成本（见 §5）。 |
| 头版引导 | **合并为单一摘要（事实）**——已删除独立斜体导语层，摘要铺满内容宽；解读统一收进标注好的「影响 · AI 分析」块（呼应 §3 铁律）。 |
| 速览形式 | **两栏密排列表（报纸"简讯栏"）**——`column` 分栏，条数随内容自由流动、不被格子绑死；每条 编号+标题+一句话+来源。 |
| 设备 | **响应式、桌面优先**；断点 820/520（次条/报头塌栏，速览两栏→单栏）。 |
| 单页 vs 多页 | **前后端分离（客户端渲染）**：每期 = 轻量壳 HTML，内容来自 `data/issues/<日期>.json`，由共享 `frontend/app.js` 客户端渲染（精读仍内联在 JSON 里）。换来的扩展性：加一期只丢一个 JSON、归档/期数切换近乎白送。**取舍**：内容依赖 JS（不开 JS 无内容、无 SEO），且不再是"可整文件分享/邮寄"的自包含页——若将来要邮件推送（M4）需另留单文件导出路径，见 §15。〔此条 2026-06 由"自包含单 HTML"改为本方案，以对齐已落地架构。〕 |
| sticky 跳转条 | 滚动高亮当前区段（IntersectionObserver）；**v1 只有 头条 / 速览 两项**。 |

### 14.1 M2 反馈交互（设计对话已定 · 入口形态＝常驻微缩图标）

## 15. 开放问题

- 每期具体条数（头条 3-5 / 速览 10-15 区间内微调）。
- 偏好档案（taste/style/seed-suggestions）的字段粒度与 digest 消化 prompt 的具体策略——M2 实现时定（§9 已定方向：增量改、分类路由、保留旧偏好）。
- 技术栈细节（库选型、目录）——用户要求最后再定。
